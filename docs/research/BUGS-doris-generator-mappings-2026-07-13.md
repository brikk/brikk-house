# brikk-sql generator: Doris function-mapping bugs (2026-07-13)

Filed from the Doris differential-probe program (bucket B/C, see
`REPORT-doris-differential-probe-2026-07-13.md`). Each row is a **cross-name mapping the
generator emits** that is semantically wrong. The registry now carries a matching
`divergent` hazard so `certify` refuses the confident-but-wrong ones as belt-and-braces,
but the **generator mapping itself should be fixed** — that is the real bug.

Severity legend:
- **P1 confident-but-wrong** — `certify` returns `ok=true`, so the wrong SQL SHIPS
  (runtime error or silently wrong result). Highest priority.
- **P2 catalog staleness** — `certify ok=true` but the emitted name is not on the live FE.
- **P3 fail-loud** — `certify` already REFUSES (UNMAPPABLE), so nothing wrong ships, but
  the mapping is broken / the target is wrong.

All reproductions: `SqlFragment("SELECT <expr>", "<duckdb|trino>").certify("doris").result.sql`.

> **RESOLUTION (2026-07-13, sql-focus worktree).** All P1 (1–6) and P2 (7) rows FIXED at
> the generator source; P3 rows 8–9 FIXED, row 10 DEFERRED (documented). Enhancements: all
> trivially-identical renames + `list_slice` arg-conversion FIXED; structural
> table-function rewrites DEFERRED. See the per-row `status` columns below.
>
> **Hazard-entry reconciliation.** Each P1/P2 `divergent` hazard existed only to guard the
> now-fixed wrong mapping, so its verdict was honestly downgraded `divergent → identical`
> (hazard text + provenance now note "generator fixed in …; re-verify against live FE") and
> the registry regenerated byte-clean (`tools/generate_hazards_registry.py`). Pair counts
> are unchanged (255 duckdb↔doris, 203 trino↔doris — entries edited in place), so
> HazardRegistryTest / HazardsRegistrySyncTest pins still hold. The `st_aswkb` entry was
> LEFT `conditionally-equivalent` (an inherent WKB binary-rendering divergence, independent
> of our mapping); its text was updated to "mapping added". Enhancement entries (gcd, lcm,
> list_position, suffix, get_current_timestamp, list_slice, is_nan) were already
> `identical`/`conditionally-equivalent`, so only their stale "fail-loud today" text was
> refreshed to "mapping added".
>
> **The lesson (why the wrong SQL shipped despite the hazards).** The P1 divergent hazards
> NEVER fired in `certify`, for TWO independent reasons: (a) the mitigation rule skips a
> hazard when the target generator has a dedicated renderer for the node's class — and every
> P1 source node (`ArrayOverlaps`, `UnixToTime`, `RegexpSplit`, `Struct`, `JSONArrayContains`,
> `JSONExtractScalar`) HAS one, which is exactly the renderer that emitted the wrong SQL; and
> (b) `certify` keys the hazard lookup on the PARSED source node's `sqlName()` (e.g.
> `ARRAY_OVERLAPS`), not the surface function name the hazard is keyed under (`LIST_HAS_ANY`),
> so the lookup missed regardless. The "belt-and-braces" divergent hazards were therefore
> pure documentation with zero protective effect — the real fix always had to be the
> generator mapping, which is what this work did.

## Missing mappings (enhancements — Doris has an equivalent, generator refuses)

These fail loud today (safe), but a mapping would extend coverage. Verified live:

- duckdb `greatest_common_divisor`→`gcd`, `least_common_multiple`→`lcm`,
  `list_position`→`array_position`, `suffix`→`ends_with`, `get_current_timestamp`→`now`,
  `st_aswkb`→`st_asbinary` — **identical / trivially-equivalent**.
  **FIXED (all).** `list_position`→`ArrayPosition`, `suffix`→`EndsWith`,
  `get_current_timestamp`→`CurrentTimestamp` (→`NOW()`) added as DuckdbParser canonical-node
  mappings; `gcd`/`lcm`/`st_aswkb` (no canonical node) added as a targeted
  `DorisGenerator.anonymousSql` rename allowlist. DorisVerifier: accepted;
  tests `enh_trivialRenames`.
- duckdb `list_intersect`→`array_intersect` (set equal, order not guaranteed),
  `list_zip`→`array_zip` (struct field naming), `row`→`named_struct` (needs synthesized
  field names). **DEFERRED** — struct/order-sensitive renames beyond the trivially-identical
  set; left as documented hazards (conditionally-equivalent), not tackled here.
- duckdb `list_slice(a,begin,END)`→`array_slice(a,start,LENGTH)` — **needs arg conversion
  (end→length)**; a naive rename would diverge.
  **FIXED.** `list_slice`→`ArraySlice` (DuckdbParser) + `DorisGenerator.dorisArraySliceSql`
  converts an integer-literal END index to a LENGTH (`END-BEGIN+1`); non-literal
  begin/end is flagged rather than diverging. DorisVerifier: accepted; test
  `enh_listSlice_endToLengthConversion`.
- trino `is_nan`→`isnan` (Doris fn is `isnan`, not `is_nan`).
  **FIXED.** `IsNan`→`ISNAN` DorisGenerator TRANSFORMS entry. DorisVerifier: accepted;
  test `enh_trivialRenames`.
- table functions: `unnest`→`LATERAL VIEW explode`, `json_each`→`explode_json_object`
  (structural rewrites, not scalar renames). **DEFERRED** (out of scope — structural
  table-function rewrites, not scalar renames).

- `date_diff` (both source dialects) maps only the `'day'` unit → `DATEDIFF(end,start)`;
