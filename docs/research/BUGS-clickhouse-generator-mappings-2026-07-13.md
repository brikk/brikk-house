# brikk-sql generator: ClickHouse function-mapping bugs (2026-07-13)

Filed from the ClickHouse differential-probe program (see
`REPORT-clickhouse-differential-probe-2026-07-13.md`). Each row is what the brikk-sql
ClickHouse **generator actually emits** (via `transpile("SELECT <expr>", "<src>", "clickhouse")`
on this branch) whose OUTPUT was then run against live ClickHouse 26.5.1.1 (chdb). The
registry now carries a matching `divergent` hazard for the ships-wrong ones, so a
certify-gated consumer refuses them (belt-and-braces) ‚Äî but the **generator mapping itself is
wrong** and should be fixed at the `renderedSql`/template source.

## STATUS (2026-07-13): generator fixes applied

Fixed at the generator source (`ClickhouseGenerator.kt`), pinned in
`ClickhouseGeneratorMappingBugsTest`, mirroring the Doris BUGS-fix template (commit
`0c9c309`). **10 of 14 rows FIXED, 4 DEFERRED.** All fixes are **live-differential-verified**
via chdb (ClickHouse **26.5.1.1** ‚Äî the exact BUGS probe version) vs DuckDB 1.5.4:
emitted SQL run in both engines, results diffed (see the harness note below).

Where the corrected mapping is result-identical, the matching hazard entry was reconciled
`divergent -> identical`: duckdb `log->log10`, `xor->bitXor`, `week->toISOWeek`,
`millisecond->(toSecond*1000+toMillisecond)`, and trino `week->toISOWeek`. The duckdb
`regexp_replace->replaceRegexpOne` fix was reconciled to `conditionally-equivalent` (common
first-match case correct, residual empty-pattern divergence stays surfaced). Registry
regenerated byte-clean; counts unchanged (duckdb‚Üîclickhouse 152: now 47 divergent / 70
identical / 35 cond-eq; trino‚Üîclickhouse 107: now 32 / 49 / 26).

**4 DEFERRED**: `round` and `bin` (P1) now have **live-verified shims** (recorded in the
rows below), but both are ClickHouse-NATIVE function names and this generator is
source-unaware ‚Äî applying the shim would regress ClickHouse‚ÜíClickHouse results, so they
await a policy call. `age` (return-type mismatch: DuckDB interval vs ClickHouse scalar)
and `to_days` (DuckDB-parser-level source ambiguity) are genuinely unmappable at the
generator. Per-row status is in the tables below.

> Note on certify policy #2: the still-divergent fixed rows (lower/upper unicode edge,
> dayofweek numbering) now have dedicated renderers, so policy #2 surfaces them as
> non-blocking WARNINGs rather than refusals ‚Äî the residual divergence is still visible.

Severity legend (verified against live ClickHouse, not docs):
- **P1 ships-wrong** ‚Äî ClickHouse ACCEPTS the emitted SQL but the result is semantically wrong
  vs the source engine. Highest priority (silent data corruption).
- **P2 invalid-name** ‚Äî the emitted function name does not exist in ClickHouse (errors at
  runtime); whether it ships depends on whether brikk's ClickHouse catalog wrongly lists it.
- **P3 shape/signature** ‚Äî emitted SQL is rejected by ClickHouse (wrong arity/operator).

## Correct mappings confirmed (documented for completeness ‚Äî do NOT change)

Verified emitting semantically-correct ClickHouse:
- duckdb `length` ‚Üí **`CHAR_LENGTH`** (code-point count ‚Äî correctly avoids the byte-count trap).
- duckdb `concat(a,b)` ‚Üí **`CONCAT(COALESCE(a,''), COALESCE(b,''))`** (replicates DuckDB's
  NULL-skip against ClickHouse's NULL-propagating concat). Trino `concat` ‚Üí `CONCAT(a,b)` (no
  wrap needed ‚Äî Trino also propagates NULL).
- duckdb `instr` ‚Üí **`POSITION`** (both case-sensitive; avoids ClickHouse `instr`=case-insensitive).
- trino `transform(a, x->‚Ä¶)` ‚Üí **`arrayMap(x->‚Ä¶, a)`** (avoids ClickHouse `transform`=value-remap).
- `regexp_replace` (trino) ‚Üí `REGEXP_REPLACE` ‚Äî correct for Trino (both replace-all).
- trino `from_unixtime` ‚Üí `fromUnixTimestamp(CAST(x AS Nullable(Int64)))`, `date_format` ‚Üí
  `formatDateTime` (format-string dialect caveat still applies), `width_bucket` ‚Üí `widthBucket`.

## Not bugs (documented)

- `ln` ‚Üí `LN` (both natural log ‚Äî correct; contrast with the `log` collision #3).
- `substr`/`substring` ‚Üí `SUBSTRING`: byte-vs-codepoint + start=0 divergence is a genuine
  semantic hazard (registry `divergent`), not a wrong *name* ‚Äî `substringUTF8` would fix
  multibyte but not the start=0 edge.

## Verification harness (2026-07-13)

expression, `chdb` 4.2.1 (embedded ClickHouse, reporting `version()` = **26.5.1.1** ‚Äî the
same build the probe program used) evaluates the generator's emitted ClickHouse SQL, and
the two results are compared on concrete inputs.

- **All 10 FIXED rows match** (equal results where a match is expected).
- The two intentionally-retained divergences reproduce exactly: `lower('ƒ∞')` DuckDB `i` vs
  ClickHouse `iÃá` (Turkish dotted-I full-fold edge), and `dayofweek('2023-01-01')` DuckDB
  `0` (Sun=0) vs ClickHouse `7` (ISO Sun=7) ‚Äî both kept as `divergent` hazards.
- The DEFERRED `round`/`bin` shims were validated against DuckDB (see their rows); they are
  correct but withheld only because the target names are ClickHouse-native (CH‚ÜíCH
  regression risk in a source-unaware generator).
