# Function gap report: duckdb / trino / doris

Generated by `tools/function_gap_report.py` (re-runnable; regenerates this file
and `brikk-sql/testResources/semantics/gap-report.json`).

## Method

For each ordered pair among the engines of record (duckdb, trino, doris),
every primary function name in the source catalog is classified:

- **A - same-name passthrough**: the name (or an alias) exists in the target
  catalog. Annotated with signature arity overlap (`arity-compatible` /
  `arity-suspect` / `arity-unknown`).
- **B - translated**: sqlglot parses the name into a typed node under the
  source dialect and the target generator renders that node specially
  (rename, TRANSFORMS entry, non-function expression form, or an explicit
  unsupported error). Each entry was render-checked by parsing a synthetic
  call (arg counts taken from catalog signatures) and generating under the
  target dialect.
- **C - rename candidates**: not A/B, but a plausible target equivalent was
  found. Evidence levels: `c1` = sqlglot alias/class knowledge (the node
  class's alternate sql_names, or a typed parse under another dialect whose
  target rendering lands on a target-catalog name); `c2` = catalog alias-set
  intersection (another alias of the same source function exists in the
  target); `c3` = lexical normalization only - **hints, not candidates**.
- **D - no target equivalent**: none of the above. These are exactly the
  functions `SqlFragment.unmappableFunctions(target)` flags at transpile time.

## Caveats (read before acting on this report)

- **Same name does not mean same semantics.** Bucket A is candidate-OK, not
  verified-OK. Null handling, argument order, collation, overflow and type
  coercion routinely differ between engines even for identically named
  functions. Verification requires semantic testing, which this report does
  not do.
- Catalogs exist only for doris, trino and duckdb. mysql / postgres /
  clickhouse catalog extraction is future work, so pairs involving those
  dialects are not analyzed here even though brikk supports them.
- Trino's TSV carries no variadic marker (e.g. CONCAT appears as a single
  flattened overload), so trino-side arity data understates accepted arity;
  some `arity-suspect` annotations with trino as target are false alarms.
- Synthetic-call parsing cannot exercise special-syntax functions (EXTRACT,
  TRIM ... FROM, etc.) or functions requiring literal arguments; such parse
  failures fall back to catalog-only evidence. Render checks use simple
  column arguments, so renderings that depend on literal argument values may
  differ in real queries.
- Bucket B render-check errors are explicit sqlglot UnsupportedError paths;
  entries marked `renderedNameNotInTargetCatalog` render to a function name
  the target catalog does not list - both are live translation issues worth
  follow-up.
- c3 lexical hints strip underscores and common prefixes (array_, list_,
  str_, ...). They are cheap string matches with no semantic backing.
